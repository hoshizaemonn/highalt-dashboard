# メンテナンス手順: シークレットローテーション（2026-07 セキュリティ強化）

所要時間: 約10分 / 利用者影響: 数分の接続断 + 全員一度だけ再ログイン
実施タイミング: 利用の少ない時間帯（クライアントへ事前連絡済みであること）

## 事前準備

- [ ] クライアントへメンテナンス連絡済み
- [ ] PR（highalt/security-hardening）がマージ・デプロイ済み
- [ ] Supabase / Vercel 両方の管理画面にログインできること

## 手順

### 1. login_attempts テーブル作成（先にやってOK・影響なし）

Supabase → SQL Editor で `scripts/sql/2026-07_login_attempts.sql` の内容を実行。

### 2. 新しい SESSION_SECRET を生成

ターミナルで:

```
openssl rand -base64 48
```

### 3. Vercel の環境変数を更新

Vercel → highalt-dashboard → Settings → Environment Variables

- `SESSION_SECRET` を手順2の値に差し替え（Production）
- ※この時点ではまだ反映されない（再デプロイで反映）

### 4. Supabase の DBパスワードをリセット

Supabase → Project Settings → Database → Reset database password

- 新パスワードを生成してコピー（この瞬間から旧接続文字列は無効になる）

### 5. Vercel の DATABASE_URL を新パスワードで更新

- `DATABASE_URL` のパスワード部分を新しいものに差し替え

### 6. 再デプロイ

Vercel → Deployments → 最新のProductionデプロイ → Redeploy

- 完了すると新シークレット・新DBパスワードが有効になる

### 7. 動作確認

- [ ] https://highalt-dashboard.vercel.app/login でログインできる
- [ ] ダッシュボードの数値が表示される（DB接続OK）
- [ ] わざとパスワードを5回間違え、ロックメッセージが出る（レート制限OK）
  - 確認後15分待つか、Supabaseで `DELETE FROM login_attempts;` を実行して解除

### 8. ローカルの後始末（星さんのMac）

- `.env` / `.env.local` の `DATABASE_URL` を新しい接続文字列に更新
  （ローカル開発をしない期間は削除しておくのが最も安全）
- 旧パスワードはどこにも残さない

## ロールバック

- DB接続エラーが出る場合: Vercelの `DATABASE_URL` の値（特にパスワード部分と
  ホスト名）を再確認して Redeploy
- ログインできない場合: `SESSION_SECRET` が Production に設定されているか確認
