**対象システム:** ハイアルチ PLダッシュボード  
**本番URL:** https://highalt-dashboard.vercel.app

## 1. 概要

本報告書は、ハイアルチ PLダッシュボードに実装されているセキュリティ対策の一覧と、その技術的な実装内容をまとめたものです。

---

## 2. 環境構成

| 項目 | 内容 |
|------|------|
| **ホスティング** | Vercel（サーバーレス） |
| **データベース** | Supabase PostgreSQL |
| **フレームワーク** | Next.js 15 + React + TypeScript |
| **ORM** | Prisma（PostgreSQL アダプタ） |
| **通信** | HTTPS（TLS 1.2以上） |

### インフラレベルのセキュリティ

- **Vercel:** DDoS防御、エッジキャッシュ、自動HTTPS証明書管理を提供
- **Supabase:** データの暗号化（at-rest / in-transit）、定期バックアップ、行レベルセキュリティ（RLS）対応

## 3. セキュリティ対策一覧

### 3.1 認証・認可

| 対策 | 実装状況 | 詳細 |
|------|----------|------|
| パスワードハッシュ化 | 実装済み | bcryptjs（コストファクター10）によるハッシュ化。平文パスワードはDBに保存しない |
| セッション署名 | 実装済み | HMAC-SHA256によるセッションCookieの署名・検証 |
| 署名検証のタイミング安全性 | 実装済み | `timingSafeEqual`（Node.js API）および定数時間比較（Edge Runtime）を使用し、タイミング攻撃を防止 |
| ロールベースアクセス制御 | 実装済み | `admin`（管理者）と`store_manager`（店長）の2ロール。APIルートごとに権限チェック |
| 人件費内訳の閲覧制限 | 実装済み | 店長は自店舗の内訳のみ閲覧可能。他店舗の内訳は非表示 |

### 3.2 セッション管理

| 対策 | 実装状況 | 詳細 |
|------|----------|------|
| HttpOnly Cookie | 実装済み | JavaScriptからのCookieアクセスを防止 |
| Secure属性 | 実装済み | 本番環境ではHTTPS接続時のみCookieを送信 |
| SameSite=Strict | 実装済み | クロスサイトリクエストでのCookie送信を完全ブロック |
| セッション有効期限 | 実装済み | 24時間で自動失効。期限切れセッションはCookie削除とともに拒否 |
| 不正セッション検出 | 実装済み | 署名不一致・期限切れ時にCookieを即座に削除しログイン画面へリダイレクト |

### 3.3 CSRF対策

| 対策 | 実装状況 | 詳細 |
|------|----------|------|
| Originヘッダー検証 | 実装済み | POST/PUT/DELETE/PATCHリクエストで`Origin`ヘッダーと`Host`ヘッダーの一致を検証 |
| SameSite Cookie | 実装済み | `SameSite=Strict`によりクロスサイトからのCookie送信を防止 |

### 3.4 ブルートフォース対策

| 対策 | 実装状況 | 詳細 |
|------|----------|------|
| レートリミット | 実装済み | IPアドレス単位で5回のログイン失敗後、15分間のロックアウト |
| ロックアウト通知 | 実装済み | ロックアウト時に残り時間をユーザーに表示 |
| 成功時リセット | 実装済み | ログイン成功時に失敗カウントをクリア |
| 古いレコードの自動クリーンアップ | 実装済み | 10分ごとに期限切れのレートリミットレコードを削除 |

### 3.5 入力検証

| 対策 | 実装状況 | 詳細 |
|------|----------|------|
| ファイルアップロード検証 | 実装済み | ファイルサイズ上限10MB、許可拡張子（.csv, .xlsx, .xls）、MIMEタイプチェック |
| パラメータ検証 | 実装済み | 各APIルートで必須パラメータの存在・型チェックを実施 |
| SQLインジェクション防止 | 実装済み | Prisma ORMによるパラメータ化クエリで全DBアクセスを実施。生SQLは使用していない |

### 3.6 セキュリティヘッダー

next.config.tsにて以下のセキュリティヘッダーを全レスポンスに付与しています。

| ヘッダー | 値 | 目的 |
|----------|-----|------|
| `X-Content-Type-Options` | `nosniff` | MIMEスニッフィング防止 |
| `X-Frame-Options` | `DENY` | クリックジャッキング防止（iframe埋め込み禁止） |
| `X-XSS-Protection` | `1; mode=block` | ブラウザのXSSフィルター有効化 |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | リファラー情報の漏洩防止 |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | HTTPS強制（HSTS、1年間有効） |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | 不要なブラウザAPIの無効化 |

### 3.7 データ保護

| 対策 | 実装状況 | 詳細 |
|------|----------|------|
| 通信暗号化 | 実装済み | Vercelが提供するTLS 1.2以上による全通信の暗号化 |
| DB接続暗号化 | 実装済み | Supabase PostgreSQLへのSSL接続を強制 |
| パスワード保護 | 実装済み | bcryptハッシュ化。平文保存なし |
| 機密情報の環境変数管理 | 実装済み | APIキー・パスワード・接続文字列はすべて環境変数で管理。ソースコードにハードコードしない |
| エラーログの適切な出力 | 実装済み | 全APIルートで`console.error`を使用し、エラーオブジェクトを適切にログ出力。クライアントには内部情報を返さない |
| Git管理除外 | 実装済み | `.env`ファイル、データベースファイル、認証情報を`.gitignore`で除外 |

## 4. 環境変数一覧

以下の環境変数がシステムの稼働に必要です。すべてVercelの環境変数設定で管理されています。

| 変数名 | 用途 | 備考 |
|--------|------|------|
| `SESSION_SECRET` | セッションCookieのHMAC署名鍵 | 最低32文字のランダム文字列。漏洩時は即座にローテーションが必要 |
| `ADMIN_INITIAL_PASSWORD` | 初回デプロイ時の管理者初期パスワード | 初回ログイン後に変更を推奨 |
| `DATABASE_URL` | Supabase PostgreSQL接続文字列 | `postgresql://user:pass@host:port/db` 形式。接続プールの上限は10に設定 |
| `NODE_TLS_REJECT_UNAUTHORIZED` | TLS証明書検証の制御 | Supabaseの自己署名SSL証明書に対応するため、開発環境では`0`に設定。本番ではPool設定の`ssl: { rejectUnauthorized: false }`で対応 |

## 5. ミドルウェアによるアクセス制御

Next.jsのミドルウェア（`proxy.ts`）で全リクエストを事前検査しています。

### アクセスフロー

```
リクエスト受信
  |
  +--> CSRF検証（POST/PUT/DELETE/PATCH）
  |      NG → 403 Forbidden
  |
  +--> パブリックパス？（/login, /api/auth/login, /api/auth/logout）
  |      はい → 通過（ログイン済みの場合は/dashboardへリダイレクト）
  |
  +--> セッションCookie存在チェック
  |      なし → 401（API）/ ログイン画面へリダイレクト（画面）
  |
  +--> セッション署名検証
  |      NG → Cookie削除 + 401 / リダイレクト
  |
  +--> セッション有効期限チェック
  |      期限切れ → Cookie削除 + 401 / リダイレクト
  |
  +--> リクエスト通過
```

## 6. 今後の推奨事項

### 優先度: 高

| 項目 | 詳細 | 期待効果 |
|------|------|----------|
| **Content Security Policy（CSP）の導入** | `next.config.ts`にCSPヘッダーを追加し、スクリプト・スタイルの読み込み元を制限 | XSS攻撃のリスクをさらに低減 |
| **セッションシークレットの定期ローテーション** | 3〜6ヶ月ごとに`SESSION_SECRET`を更新する運用ルールの策定 | 万が一の漏洩時の影響範囲を限定 |
| **監査ログの永続化** | アップロードログに加え、ログイン/ログアウト/設定変更のログをDBに保存 | 不正アクセスの事後調査を可能に |

### 優先度: 中

| 項目 | 詳細 | 期待効果 |
|------|------|----------|
| **二要素認証（2FA）の導入** | TOTP（Google Authenticator等）による二段階認証 | パスワード漏洩時のリスク軽減 |
| **パスワードポリシーの強化** | 最低文字数・英数字混合・記号必須などのルール | 弱いパスワードの排除 |
| **セッションの永続化（Redis等）** | 現在のインメモリレートリミットをRedis等に移行 | サーバーレス環境でのレートリミットの信頼性向上 |
| **IPアドレスベースのアクセス制限** | 社内ネットワークまたはVPNからのみアクセス許可 | 不正アクセスのリスク低減 |

### 優先度: 低

| 項目 | 詳細 | 期待効果 |
|------|------|----------|
| **脆弱性スキャンの定期実施** | npm audit、Snyk等によるパッケージ脆弱性チェックの自動化 | サプライチェーン攻撃の予防 |
| **ペネトレーションテスト** | 外部専門業者によるセキュリティ診断 | 未知の脆弱性の発見 |
| **バックアップ・リカバリ手順の文書化** | Supabaseのバックアップからの復旧手順を明文化 | 障害時の復旧時間短縮 |

## 7. まとめ

本システムでは、認証・認可・セッション管理・CSRF対策・ブルートフォース対策・入力検証・セキュリティヘッダー・データ保護の各領域にわたり、Webアプリケーションに求められる標準的なセキュリティ対策を実装済みです。

今後は、CSPの導入やセッション管理の外部化、監査ログの充実を段階的に進めることで、さらなるセキュリティレベルの向上を図ることを推奨します。
